# 移动端图标化升级选项设计

## 问题

当前升级面板的三个按钮只有居中小号文字。竖屏实测时，玩家很难快速区分新武器、已有武器升级和属性功法，也看不到本次选择会到达哪一级。面板还保留较多无效空白，使有效信息在手机上显得更小。

本轮只优化选择信息层级和触控扫读，不改升级池、升级概率、数值或暂停流程。

## 方案比较

1. **现有按钮图标化（采用）**：继续使用三个大按钮，在左侧接入正式图标，文字改为类别、进度、名称和效果摘要。复用现有美术并保持点击逻辑不变，风险最低。
2. **重做自定义卡片树**：每项拆成图标、多个 Label 和边框节点，布局自由但场景复杂、触摸区域和响应式适配需要重新验证。
3. **仅放大文字并压缩空白**：改动最少，但仍无法快速识别升级类型和当前等级。

## 数据

- `data/upgrades/core_upgrades.json` 为每个升级定义显式 `icon_path`。
- 武器解锁与武器精进复用已有正式武器图标；属性与组合功法复用现有飞剑、罗盘、灵石、符纸等图标。
- `UpgradeSystem.get_choices()` 在返回选项时补充：
  - `category_label`：`新武器`、`武器精进`、`功法强化` 或 `组合功法`。
  - `progress_label`：解锁显示 `Lv.1`；武器精进显示 `Lv.N→N+1`；功法显示本次将达到的 `第N重`。
- 进度只根据当局 `owned_weapons` 与 `upgrade_stacks` 计算，不写回配置数据。

## 面板

- 保留三个 `Button` 作为完整触控区域，最小高度从 104 提升到 116 逻辑像素。
- 图标固定在左侧，最大宽度 68 逻辑像素，并启用等比缩放。
- 每项文字固定为三行：`类别 · 进度`、升级名称、效果摘要。缺少摘要时只显示前两行。
- 文字左对齐，字号提升到 17；长摘要使用裁切而不是撑大按钮。
- 面板减少底部空白，但保持标题、战斗暂停提示、三个按钮和 12 像素间距。
- `icon_path` 缺失或资源加载失败时，按钮必须清除旧图标并继续显示文字；复用按钮时不能残留上一组选项的图标。

## 结构

- `UpgradeSystem` 负责从运行状态计算类别与进度，保持展示语义与升级规则一致。
- `UpgradeChoicePanel` 只格式化文字、缓存并加载纹理、绑定按钮，不推断武器等级。
- JSON 数据负责升级与正式图标的绑定，新增武器时无需修改面板代码。
- 不新增运行时节点类型，不改变 `upgrade_selected` 信号和 `GameLoop` 的暂停所有权。

## 验证

- `test_upgrade_system.gd` 覆盖四种类别标签、武器等级进度和功法层数进度。
- `test_upgrade_choice_panel.gd` 覆盖三行文本、图标加载、触控尺寸、左对齐、图标宽度、缺图回退和按钮复用时清理旧图标。
- `test_game_database.gd` 确认所有核心升级的 `icon_path` 指向可加载纹理。
- 运行完整 Godot 测试、600 帧场景烟测，并在 390x844 竖屏窗口检查第一次升级：三种类型能快速区分，文字不重叠，三个选项均可点击。

## 成功标准

- 不修改升级概率、升级数值、升级池和战斗暂停逻辑。
- 每个核心升级都有正式可加载图标。
- 第一次升级可直接看出新武器与功法的区别，武器升级显示前后等级。
- 任何无效图标路径都不会导致解析错误、空白选项或旧图标残留。
