# 模块化武器与首批八把武器设计

## 1. 目标与依据

本子项目落实 `2026-07-12-forty-hour-content-expansion-design.md` 的第 6.4、14.1、16.1 和 17 节，将当前仅支持投射物与瞬时脉冲的武器实现扩展为可承载 16 把正式武器的受控效果管线。

本阶段完成后，游戏必须具备：

- 四个流派各两把正式可用武器，共八把。
- 投射物、范围、轨道、召唤物四类载体。
- 灼烧、冻结、破甲、封印四种可感知状态。
- 所有武器共用目标选择、命中、状态、反馈和对象池契约。
- 玩家现有存档、四武器槽、升级选择和现有五把武器行为保持可用。
- 后续增加光束和陷阱时不需要修改 `GameLoop` 的武器类型分支。

该子项目不是简单增加三个 JSON 文件。验收重点是让武器之间产生不同走位、目标选择与组合收益，同时维持高密度战斗下的动作一致性和移动端预算。

## 2. 非目标

- 本阶段不实现武器进化、被动法宝和角色主动技能。
- 本阶段不实现光束与陷阱载体；它们在后续八把武器中接入同一管线。
- 本阶段不建立任意节点连接的效果图编辑器，也不允许在数据中保存可执行脚本。
- 本阶段不制作完整章节解锁流程；八把武器先进入开发构筑池，正式解锁条件由进度系统子项目接管。
- 本阶段不通过增加生命或伤害数值来伪造武器差异。

## 3. 方案选择

采用受控模块管线：

`触发 -> 目标选择 -> 载体 -> 命中包 -> 状态规则 -> 视觉与音频反馈`

每一层只接受已注册的模块 ID 和有边界的数据结构。拒绝以下两种实现：

- 通用效果图解释器：组合自由度过高，非法组合、调试成本和移动端性能风险不可控。
- 每把武器独立脚本：首批开发快，但会重复实现命中、状态、对象池与反馈，无法稳定扩展到 16 把武器。

## 4. 系统边界

### 4.1 WeaponRuntime

现有场景节点名 `WeaponSystem` 保留，避免破坏场景和测试；其职责收敛为武器运行时：

- 持有最多四把武器的等级、冷却和分支状态。
- 根据定义和当前等级生成不可变的 `effect_request`。
- 不查找敌人、不生成场景、不直接造成伤害。
- 不包含 `projectile`、`pulse` 等载体类型分支。

### 4.2 CombatEffectPipeline

新增独立节点，负责顺序执行一个效果请求：

1. 使用目标模块从显式传入的战斗上下文中选出目标或方向。
2. 将请求交给已注册载体。
3. 载体在命中时创建统一 `hit_packet`。
4. `HitResolver` 结算伤害、击退和状态。
5. 发出命中与状态信号，供特效、音效、震动和统计监听。

管线不直接读取 `WeaponRuntime` 内部状态。`GameLoop` 仅提供场景根节点、玩家位置、候选目标、对象池和本局时间，不再判断武器类型。

管线执行后返回 `executed`、`no_target`、`pool_queued` 或 `invalid_request`。`no_target` 不消耗完整冷却，运行时在 0.1 秒后重试；`pool_queued` 保留原请求但不重复扣除冷却；`invalid_request` 在开发构建中直接报错并停止该武器。

### 4.3 TriggerRegistry

一个武器包含一至三个有稳定 `effect_id` 的线性效果，每个效果独立选择触发、目标、载体和命中包。首批注册：

- `periodic`：按冷却周期请求一次效果。
- `persistent`：装备时生成，卸下或本局结束时释放；用于轨道载体。
- `on_player_hit`：玩家实际失去生命后触发，拥有独立冷却；用于剑葫芦反击。
- `on_kill`：收到该武器归因的击杀事件后触发，为后续升级预留并在本阶段完成注册测试。

玩家受击和击杀通过显式战斗信号传入，触发模块不能自行查找玩家生命节点或扫描结算统计。效果列表不是任意图：效果之间不能引用彼此，也不能形成循环。

### 4.4 TargetSelector

首批注册五种目标模块：

- `nearest`：范围内最近敌人。
- `lowest_health`：范围内生命比例最低的敌人，距离作为稳定次序。
- `sector`：面向目标方向的扇区内敌人。
- `radial`：以玩家为中心生成均匀方向，不要求锁定目标。
- `self`：以玩家位置为载体中心。

相同输入和固定种子必须得到相同结果；目标失效时请求安全结束，不生成无方向载体。

### 4.5 CarrierRegistry

首批四类载体均为明确场景与脚本，不使用一个包含大量分支的万能载体：

- `projectile`：直线、扇射、放射、穿透和命中爆炸。
- `area`：瞬时脉冲或有明确持续时间的地面范围；同一目标遵守命中间隔。
- `orbit`：围绕玩家运行，按单目标命中冷却结算，不因帧率重复伤害。
- `summon`：有寿命的自动寻敌单位，拥有明确攻击间隔，不与敌人实体碰撞规则混用。

四类载体全部接入 `PoolService`。池释放必须清空命中记录、状态引用、视觉覆盖和计时器。

### 4.6 HitResolver

所有玩家武器伤害必须通过统一命中包：

```text
source_weapon_id
source_instance_id
base_damage
damage_tags
knockback
hit_position
status_payloads
hit_effect_id
```

结算顺序固定为：状态前置修正、直接伤害、击退、状态施加、反应、命中反馈。范围爆炸和召唤物攻击不能绕过该顺序直接调用 `HealthComponent.take_damage()`。

### 4.7 StatusController

每个可受武器影响的敌人挂载一个状态控制器。状态控制器只通过公开查询影响移动和动作状态，不直接改写动画、碰撞层或伤害有效帧。

- 冻结发生在前摇、有效帧或恢复期间时，暂停该状态剩余时间；解除后从原阶段继续，不能跳过有效帧或死亡。
- 封印只阻止新的特殊动作进入前摇，不中断已经开始的攻击；基础移动和接触攻击仍可进行。
- 死亡开始后拒绝新状态并立即停止状态计时与视觉。

## 5. 状态与反应

### 5.1 灼烧

- 最多三层，每层独立贡献持续伤害。
- 重复施加刷新持续时间但不越过层数上限。
- 持续伤害通过 `HitResolver` 的状态伤害入口记录来源，不触发普通投射物命中特效。

### 5.2 冻结

- 寒意达到三层后进入短时冻结，寒意清零。
- 冻结结束后获得短时冻结抗性，期间只能重新积累一层寒意。
- Boss 不完全停止，而是降低动作计时速度；阶段事件和死亡不受影响。

### 5.3 破甲

- 单层状态，使后续直接武器伤害提高 15%。
- 重复施加只刷新持续时间，不无限叠加。
- 持续伤害不享受破甲，避免灼烧在高密度战斗中失控放大。

### 5.4 封印

- 阻止敌人开始冲锋、远程施法、召唤和支援动作。
- 不打断已经进入前摇的动作，也不关闭当前有效帧。
- 对没有特殊动作的敌人仍显示短时封印标记，但不附加额外减速或伤害。

### 5.5 冰火反应

灼烧施加到冻结目标，或冻结在灼烧目标上完成时，触发一次“冷热崩裂”：

- 消耗全部灼烧与冻结状态。
- 造成一次有上限的固定反应伤害。
- 触发独立的青白碎裂效果，不使用红色高威胁预警色。
- 同一目标每 0.75 秒最多触发一次，防止高射速组合重复结算。

首批只实现这一种跨状态反应，用来证明管线可组合；更多反应在雷印和墨毒咒接入时扩展。

## 6. 首批八把武器

| 流派 | 武器 | 目标 | 载体 | 核心玩法 | 状态与等级方向 |
| --- | --- | --- | --- | --- | --- |
| 御剑 | 飞剑 | 最近 | 投射物 | 稳定追击、升级后穿透和双剑夹击 | 无状态，强调直接伤害与穿透 |
| 御剑 | 剑葫芦 | 自身 | 轨道 | 靠近敌群获得高覆盖，扩大轨道或提高转速形成走位取舍 | 高等级受击后释放一次有冷却的反击剑雨 |
| 符法 | 符火 | 最近 | 投射物 | 命中爆炸，适合密集敌群 | 施加灼烧，升级扩大爆炸或增加火花 |
| 符法 | 寒霜符 | 扇区 | 投射物 | 扇形霜符控制接近路线 | 积累寒意并冻结，升级增加扇面或穿透冰刺 |
| 机关 | 机关连弩 | 最低生命 | 投射物 | 高频收割与直线贯穿 | 高等级附加破甲，散射与重弩方向后续由分支系统承接 |
| 机关 | 灵针阵 | 放射 | 投射物 | 近身清场和高数量弹幕 | 高等级附加短时破甲，保持单发伤害低 |
| 镇魂 | 封魔铃 | 自身 | 范围 | 周期声波击退，创造可读安全空间 | 高等级提高封印概率与持续时间 |
| 镇魂 | 引魂灯 | 最近 | 召唤物 | 生成有寿命魂火，自动追击不同目标 | 强调数量与持续时间，不额外叠加控制状态 |

每把武器必须拥有至少一个改变空间使用方式的升级，而不只是伤害或冷却变化。首批仍使用五级结构；武器分支与进化由后续子项目读取同一等级数据扩展。

## 7. 数据结构

每个武器定义使用版本化结构。武器包含一至三个独立线性效果，核心字段如下：

```json
{
  "id": "talisman_fire",
  "version": 1,
  "school": "talisman",
  "effects": [{
    "effect_id": "fire_bolt",
    "trigger": { "id": "periodic", "cooldown": 1.15 },
    "target": { "id": "nearest", "range": 360 },
    "carrier": { "id": "projectile", "speed": 440, "count": 1 },
    "hit": {
      "damage": 9,
      "splash_radius": 72,
      "statuses": [{ "id": "burn", "stacks": 1, "duration": 3.0 }]
    }
  }],
  "visual": {
    "carrier": "talisman_fire_projectile",
    "hit": "talisman_fire_burst"
  },
  "levels": []
}
```

所有模块 ID、数值范围、视觉引用和等级覆盖字段在数据库加载时验证。未知模块、缺失视觉、负冷却、零寿命、无效状态或重复等级必须使开发构建加载失败。

等级覆盖通过 `effect_id` 和字段白名单定位，不能使用任意字典路径。删除或重命名效果时，数据验证必须同时更新所有等级引用。

现有五把武器在本子项目中一次性迁移到版本 1。运行时代码不长期保留旧字段兼容分支，避免两套格式共同演进。存档只保存稳定武器 ID、等级和分支选择，不保存定义副本；现有五个武器 ID 保持不变，因此旧存档无需内容迁移即可读取新定义。无法识别的新字段由存档版本迁移层忽略，无法识别的武器 ID 记录错误并跳过该槽位。

## 8. 动作一致性

- 武器触发与敌人动作状态相互独立，状态只能通过公开契约改变动作计时或是否允许进入特殊动作。
- 载体命中不能直接切换敌人状态机；硬直请求由敌人状态机判断是否接受。
- 冻结和封印不能跳过前摇、有效帧、恢复或死亡中的任一阶段。
- 召唤物攻击也必须拥有攻击间隔与命中时刻，不允许每帧接触伤害。
- 所有高伤范围效果先显示非红色玩家特效；敌方红色高威胁预警始终在视觉层级上方。

## 9. 正式美术

本阶段补齐以下正式资产：

- 剑葫芦：武器图标、葫芦主体、轨道飞剑、反击剑雨和命中特效。
- 寒霜符：武器图标、霜符投射物、寒意标记、冻结覆盖和冰火崩裂。
- 引魂灯：武器图标、灯体、魂火召唤物、魂火攻击和消散效果。
- 封魔铃：声波范围、封印标记和命中反馈。
- 通用状态：灼烧、破甲、封印的手机尺寸可读标记。

使用内置 image-2 生成纯绿幕源图，源图进入 `art/source_green`，抠图成品进入对应武器与状态目录。每批生成接触表，并在 720x1280 竖屏战斗背景上检查轮廓。玩家武器特效不得使用敌方预警的纯红主色。

## 10. 移动端预算

- 活跃投射物继续受 250 上限约束。
- 轨道载体每把最多 8 个可见实体；同一目标命中检测不高于每秒 10 次。
- 召唤物每把基础上限 6 个，进化前总上限 12 个。
- 持续范围每把同时最多 6 个；离屏范围优先降低粒子，不停止伤害计时。
- 状态视觉使用单个聚合节点显示层数，不按层生成独立粒子节点。
- 对象池满时请求进入短队列或安全取消非关键视觉，不能跳过已经发生的伤害动作。

## 11. 测试策略

### 11.1 数据验证

- 八把武器定义全部加载，ID 唯一且引用存在。
- 所有模块 ID 在注册表中，数值和等级覆盖字段合法。
- 每把武器都存在对应解锁和等级升级定义。

### 11.2 单元测试

- 冷却、等级覆盖和四槽限制。
- 五种目标选择器的范围、稳定次序和空目标行为。
- 四类载体的寿命、命中间隔、穿透、释放与池重置。
- 命中顺序、破甲修正、状态层数、刷新、免疫和冰火反应冷却。
- 冻结与封印在前摇、有效帧、恢复和死亡阶段不破坏动作顺序。

### 11.3 集成与模拟

- 八把武器各执行固定时间，至少产生一次合法目标、载体、命中和反馈记录。
- 四武器组合运行三分钟，确认对象池回收、状态上限和无重复命中失控。
- 固定种子比较八把武器的攻击形态签名，不能退化成相同的投射物参数差异。
- 完整 Godot 测试、场景烟测、对象池 churn 和遭遇模拟保持通过。

### 11.4 视觉验收

- 在 720x1280 和 1080x1920 竖屏检查四种载体与四种状态可辨识。
- 高密度敌群中仍能区分敌方红色预警、玩家武器和经验拾取物。
- 新资产无绿边、锚点漂移、透明脏边或过度遮挡。

## 12. 验收标准

只有同时满足以下条件，本子项目才完成：

- 数据库加载并验证八把正式武器，四流派各两把。
- 实际战斗中出现投射物、范围、轨道和召唤物四种载体。
- 灼烧、冻结、破甲、封印及冰火崩裂均具有规则、反馈和测试。
- `GameLoop` 不再按武器类型执行投射物或脉冲分支。
- 所有玩家武器伤害经由统一 `HitResolver`。
- 冻结、封印和死亡状态通过动作一致性测试。
- 八把武器均能在升级池中解锁、升级，并受四槽上限约束。
- 新增正式美术完成绿幕源图、透明成品、接触表和竖屏实机场景检查。
- 完整自动测试、三分钟确定性模拟、对象池 churn 和 Godot 场景烟测无错误。
- 现有五把武器的可用行为和中文显示无回归。

## 13. 实施顺序

1. 数据结构、模块注册表与数据库验证。
2. `WeaponRuntime` 请求生成与现有五把武器迁移。
3. `HitResolver`、`StatusController` 和动作状态契约。
4. `CombatEffectPipeline` 与投射物、范围载体迁移。
5. 轨道、召唤载体及对象池接入。
6. 剑葫芦、寒霜符、引魂灯的数据、升级和正式美术。
7. 八武器集成、确定性模拟、视觉检查与平衡首轮。

每一步使用测试先行并形成可独立回滚的提交，不在管线未稳定前批量接入新素材。
